# Philadelphus — перечень задач на исправление уязвимостей

Анализ проведён по исходному коду решения (713 .cs-файлов). Изменения в код **не вносились**.
Ниже — 10 наиболее критичных проблем безопасности, отсортированных по приоритету, с указанием места,
сути риска и рекомендации по исправлению.

## 1. 🔴 Хардкод учётных данных и реальных строк подключения в репозитории
**Где:**
- `Philadelphus.Presentation.Wpf.UI/App.config` — 5 строк подключения PostgreSQL с паролем `Dniwe2002`, включая публичные хосты (`212.22.94.51`, `185.65.200.75`).
- `Philadelphus.Infrastructure.Persistence.ADO.PostgreSQL/Context.cs` — захардкоженная строка `Host=localhost;Username=postgres;password=Dniwe2002;...`.
- `Philadelphus.Core.Domain/Configurations/RedisOptions.cs` — `Password = "philapass123"` по умолчанию.
- `Philadelphus.Presentation.Wpf.UI/appsettings.json` — Redis `"Password": "philapass"`.

**Риск:** один и тот же пароль переиспользуется на всех окружениях, включая доступные из интернета БД. Любой, у кого есть доступ к репозиторию (или к собранному `App.config`), получает прямой доступ к продакшен-БД. CWE-798, CWE-259.

**Что сделать:**
- Немедленно **сменить (ротировать)** все скомпрометированные пароли БД и Redis на всех хостах.
- Удалить секреты из кода и из истории git (`git filter-repo`/BFG); добавить `App.config`/`appsettings.json` с секретами в `.gitignore`, оставив в репозитории только шаблоны (`.example`).
- Перевести секреты на переменные окружения / User Secrets / защищённое хранилище (DPAPI, Windows Credential Manager, Vault).
- Назначить **уникальные** учётные данные на каждое окружение.

## 2. 🔴 Выполнение произвольного кода через загрузку плагинов/расширений
**Где:** `Philadelphus.Core.Domain.ExtensionSystem/Services/ExtensionManager.cs` (`LoadExtensionsAsync`) — `Assembly.LoadFrom(dll)` для **любого** `*.dll` из каталога плагинов, затем `Activator.CreateInstance` для всех типов `IExtensionModel`. Каталоги заданы в `appsettings.json` → `PluginsDirectoriesStrings`, среди них **относительный** путь `.\Plugins`.

**Риск:** любой `.dll`, помещённый в `.\Plugins` (рабочий каталог) или в `%USERPROFILE%\...\Plugins`, выполняется в процессе приложения с правами пользователя — классический DLL-planting → RCE. Контроль отсутствует, в отличие от C#-адаптера (см. ниже), где есть проверка SHA-256. CWE-94, CWE-427.

**Что сделать:**
- Применить ту же модель доверия, что в `CSharpAssemblyAdapter`: обязательный all-list по SHA-256 / проверка строгой подписи (Authenticode) перед загрузкой.
- Убрать относительные пути плагинов; использовать только защищённый от записи каталог.
- Рассмотреть загрузку в изолированный `AssemblyLoadContext` с возможностью выгрузки и минимальными правами.

## 3. 🟠 Секреты БД/Redis хранятся в открытом виде на диске
**Где:** `appsettings.json` → `ConfigurationFilesPathesStrings` указывает на `connection-strings-config.json` в `%USERPROFILE%\AppData\Local\Philadelphus\Configuration\`; `ConnectionStringsCollectionConfig`/`ConnectionStringsContainer` сериализуются как обычный JSON.

**Риск:** строки подключения с паролями лежат в открытом виде в профиле пользователя; доступны любому процессу пользователя и попадают в бэкапы. CWE-312, CWE-256.

**Что сделать:** шифровать секреты при хранении (DPAPI `ProtectedData` с областью CurrentUser, либо Windows Credential Manager). Хранить в конфиге только ссылку на запись хранилища, а не сам пароль.

## 4. 🟠 Отсутствие принудительного TLS для PostgreSQL
**Где:** строки подключения в `App.config` содержат только `Timeout`/`CommandTimeout`, без `SslMode=Require`/`Trust...`; хосты включают публичные IP.

**Риск:** соединение с удалёнными/публичными БД может идти открытым текстом — перехват учётных данных и данных. CWE-319.

**Что сделать:** задать `SslMode=VerifyFull` (или `Require` минимум) и корректную проверку сертификата для всех удалённых подключений; запретить незашифрованные соединения на уровне сервера БД.

## 5. 🟠 Kafka без аутентификации и шифрования
**Где:** `Philadelphus.Infrastructure.Messaging.Kafka/KafkaConsumer.cs` — `ConsumerConfig` задаёт только `BootstrapServers`/`GroupId`/`AutoOffsetReset`; нет `SecurityProtocol`, `SaslMechanism`, SSL.

**Риск:** при доступности брокера в сети — неаутентифицированное чтение и инъекция сообщений (`SecurityProtocol` по умолчанию `Plaintext`). CWE-306, CWE-319.

**Что сделать:** настроить `SecurityProtocol=SaslSsl`, SASL-аутентификацию и TLS; вынести параметры в защищённую конфигурацию.

## 6. 🟠 Redis: слабый пароль по умолчанию, без TLS
**Где:** `RedisOptions.cs` (`philapass123`), `appsettings.json` (`philapass`); Redis используется как распределённый кэш (`DistributedPhiladelphusRepositoryContentCache`).

**Риск:** предсказуемый пароль и отсутствие TLS → доступ к кэшу, его отравление/раскрытие данных, если Redis доступен по сети. CWE-1188, CWE-319.

**Что сделать:** убрать дефолтный пароль (делать конфигурацию обязательной), включить TLS, ограничить сетевой доступ к Redis.

## 7. 🟡 SQLite-отчёты: выполнение хранимого SQL и ATTACH произвольных файлов
**Где:** `Philadelphus.Infrastructure.Persistence.EF.SQLite/Repositories/SqliteEfReportsInfrastructureRepository.cs` — `ExecuteReportAsync`/`RefreshMaterializedViewAsync` читают `creation_sql` и `attach_databases` из таблицы `system_report_metadata` и выполняют их (`CREATE TABLE ... AS SELECT`, `ATTACH DATABASE @path AS ...`).

**Риск:** идентификаторы и путь параметризуются/квотируются (это хорошо), но **содержимое `creation_sql` и список ATTACH берутся из таблицы БД и исполняются**. Если файл БД общий/редактируемый или подменён, возможно выполнение произвольного SQL и подключение произвольных файлов БД (раскрытие данных). CWE-89 (доверие к данным как к коду).

**Что сделать:** не хранить и не исполнять «сырой» SQL из метаданных; перестраивать представления из верифицированного описания. Ограничить ATTACH белым списком каталогов и проверять канонизированный путь. Открывать отчётную БД в режиме только для чтения там, где это применимо.

## 8. 🟡 Утечка чувствительных данных в логи и подробные ошибки БД
**Где:** `SqliteEfReportsInfrastructureRepository.cs` логирует полный SQL, имена отчётов и пути ATTACH (`_logger.Information($"...{recreateSql}")`, `...{report.AttachDatabases}`). В строках подключения `App.config` включено `Include Error Detail=true`.

**Риск:** в логи попадают SQL-запросы, схема и значения; `Include Error Detail=true` добавляет в исключения значения данных — раскрытие информации. CWE-209, CWE-532.

**Что сделать:** отключить `Include Error Detail=true` в продакшене; не логировать полные запросы/пути на уровне Information; маскировать чувствительные значения; пересмотреть уровни логирования.

## 9. 🟡 Скрипт ослабляет защиту Windows Defender
**Где:** `windows_defender_prepare.bat` — `powershell -ExecutionPolicy Bypass` и `Add-MpPreference -ControlledFolderAccessAllowedApplications ...` (внесение приложения в исключения защиты от шифровальщиков), с захардкоженными путями `D:\MelSV_Projects\...`.

**Риск:** ослабляет защиту узла (Controlled Folder Access — антиransomware), приучает пользователя запускать скрипты с обходом политики выполнения. CWE-693.

**Что сделать:** убрать из поставки или заменить на документированную инструкцию; не отключать защиту по умолчанию; при необходимости — подписанный установщик с явным согласием пользователя и корректными путями.

## 10. 🟡 Загрузка доверенных сборок в основной процесс без изоляции
**Где:** `Philadelphus.Infrastructure.AssemblyAdapters.CSharp/CSharpAssemblyAdapter.cs` (`Assembly.LoadFrom` + `Activator.CreateInstance`). Проверка SHA-256 есть, но сборки исполняются в основном процессе с полными правами; конструкторы сторонних типов вызываются немедленно.

**Риск:** даже при allow-list нет разделения привилегий/песочницы; ошибка в управлении списком хэшей или его обход (как в п.2) ведёт к выполнению кода с правами приложения. Архитектурно отсутствует наименьшая привилегия. CWE-250, CWE-265.

**Что сделать:** унифицировать модель доверия для всех адаптеров и расширений; выполнять сторонний код в изолированном `AssemblyLoadContext`/отдельном процессе с ограниченными правами; не вызывать конструкторы до валидации типа.

### Сводные рекомендации
1. Срочно: ротация всех паролей (п.1), затем чистка истории git.
2. Закрыть RCE-вектор плагинов (п.2) до следующего релиза.
3. Перевести все секреты в защищённое хранилище и включить TLS для PostgreSQL/Redis/Kafka (п.3–6).
4. Пересмотреть доверие к данным-как-коду в отчётах и логирование (п.7–8).
