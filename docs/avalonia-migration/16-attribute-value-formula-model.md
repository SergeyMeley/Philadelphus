# 16. Чистая модель значения атрибута: формула — источник истины

Проектная проработка перехода от дуального хранения (`Value` + `ValueFormula`) к модели,
где **`ValueFormula` — единственный канонический вход**, а `Value` — производный кеш.
Контекст: баг «обнуления» унаследованного значения при ручном вводе (карточка #66708083 и
далее) вскрыл конфликт двух записываемых полей и флага переопределения. Чистое решение —
убрать дуальность.

## 1. Целевая модель

- **`ValueFormula`** — единственное, что задаёт пользователь и что хранится канонически.
- **`Value`** — материализованный результат вычисления формулы. **Кеш**, нужен отчётам и
  экспорту. Пишет его ТОЛЬКО пересчёт; UI/таблицы `Value` напрямую не трогают никогда.
- **«Переопределено»** = локальная `ValueFormula` ≠ родительской. Один источник, конфликт
  `Value`/`ValueFormula` исчезает (с ним — и класс багов вокруг `_isValueOverridden`).
- Вычисление, граф зависимостей, материализация `Value`, режимы пересчёта — **бизнес-логика
  в Core** (доменный сервис). Presentation — только отображение и триггер правки.

## 2. Правила хранения формулы (что именно пишем)

Пользователь задаёт значение разными способами; храним **ровно как задано**, без
принудительной нормализации:

| Ввод | Хранимая `ValueFormula` |
|------|--------------------------|
| Выбор из выпадающего списка | `=[uuid]` |
| Введённая формула (`=[uuid]`, `=ЛИСТ("Мой лист")`, `="Мой лист"`, …) | дословно, как ввёл пользователь |
| Системный базовый тип, «голое» значение без `=` (`12000`, `abc`) | литерал-формула по типу: `=12000`, `="abc"`, дата/время — в формате типа |
| Пользовательский тип, «голое» значение без `=` | **отклоняем** (только пик или формула; ввод формул НЕ запрещаем) |

`Value` всегда = результат вычисления `ValueFormula`. `=[uuid]` резолвится в лист; литерал —
через `TrySetSystemBaseValueFromString` (find-or-create листа), результат — в кеш `Value`.

## 3. Отображение и редактирование

- **Просмотр ячейки (обе таблицы):** всегда вычисленное значение (или код ошибки формулы).
- **Строка формул:** дословная `ValueFormula` (тут пользователь видит/правит сырую формулу).
- **Редактирование ячейки значения:**
  - системный базовый тип — текстовый ввод, показываем **вычисленное** значение; «голый»
    ввод оборачиваем в литерал; ввод с `=` — формула дословно;
  - пользовательский тип — пик из списка (→ `=[uuid]`) или формула с `=`; «голый» текст как
    простое значение **не принимаем**.
- **Мгновенность:** правленый атрибут пересчитывается СРАЗУ и синхронно, независимо от режима
  пересчёта (Manual/Auto/AutoCurrentElement). У `=[uuid]` зависимостей нет — резолвинг тривиален.

## 4. Архитектурный сдвиг

Уже в Core: `FormulaAstEvaluator`, `FormulaResult`, `FormulaExecutionContext`
(`Philadelphus.Core.Domain.FormulaEngine`).

Переезжает из Presentation (`RepositoryFormulaBarVM`) в новый доменный сервис Core:
- `RecalculateFormulaAttribute` (рекурсия + цикл-детект + порядок зависимостей);
- `TryApplyFormulaResult` (материализация результата в `Value`);
- `CreateFormulaExecutionContext`, `GetReferencedFormulaAttributes`, `FormatFormulaErrorCode`;
- режимы пересчёта.

Presentation после переезда: строка формул и таблицы дёргают доменный сервис и показывают
`Value`/код ошибки. `AttributeValueText` сводится к «обернуть ввод в формулу (пик/литерал/
дословно) → вызвать пересчёт атрибута».

## 5. Шаги (с билдом на стороне пользователя между шагами)

1. **Core-сервис пересчёта.** Новый `IAttributeFormulaService` (+ реализация) в Core: переносим
   туда оркестрацию из `RepositoryFormulaBarVM`. VM делегирует, поведение не меняем (чистый
   рефактор-переезд). DI-регистрация.
2. **Запись только формулой.** `AttributeValueText`/`EditText`-путь/WPF-сеттер перестают писать
   `Value`. Пик → `=[uuid]`, системный «голый» ввод → литерал, формула → дословно, «голый» текст
   для пользовательского типа → отказ. После записи — синхронный пересчёт атрибута через сервис.
3. **Переопределение по формуле.** `_isValueOverridden`/`GetEffectiveValue` → сравнение
   `ValueFormula`. Удаляем текущий workaround (пере-присвоение `Value`).
4. **UX отображения/редактирования.** Обе таблицы + строка формул по правилам раздела 3
   (системный — текст вычисленного; пользовательский — пик/формула, без «голого» текста).
5. **Миграция данных.** Существующие строки (`Value` задан, формула пуста) → `=[uuid]`
   (пользовательский тип) / `=<литерал>` (системный). Делаем раньше.
6. **Тесты** на каждый шаг (узлы: запись формулы, материализация `Value`, переопределение,
   мгновенный пересчёт, миграция).

## 6. Открытые вопросы / риски

- **Синтаксис литералов.** Обёртка системного «голого» ввода должна давать валидную формулу,
  которую понимает парсер (строки — в кавычки с экранированием; даты/время — в ожидаемом
  формате). Уточнить токенайзер/парсер при шаге 2.
- **Производительность.** `Value` остаётся кешем (не вычисляем в геттере), отчёты читают кеш —
  деградации быть не должно. Полный пересчёт нужен на загрузке/миграции.
- **Загрузка.** `Value` персистится как кеш; на загрузке либо доверяем кешу, либо прогоняем
  пересчёт (решить на шаге 5 вместе с миграцией).
- **Границы Core↔Presentation.** Уведомления (`INotificationService`) уже доступны в Core —
  сервис сможет слать предупреждения о типах/циклах без обратной зависимости на Presentation.
