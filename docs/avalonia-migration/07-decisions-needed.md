## D1 — Avalonia XPF: рассматривать как альтернативу или нет?

**Статус:** Открытый

**Контекст:** Avalonia XPF — коммерческое решение от Avalonia-команды, позволяющее запускать существующий WPF-код на Avalonia-рантайме с минимальными изменениями. Это "страховочный путь", который резко снижает стоимость миграции, но создаёт зависимость от платного продукта.

**Варианты:**
- **A (рекомендуемый default):** Не рассматривать XPF. Выполнять чистую миграцию на open-source Avalonia. Больше усилий, лучший конечный результат.
- **B:** Провести spike XPF параллельно с Этапом 1. Если XPF позволяет запустить WPF-код с 80%+ функциональности — использовать как промежуточный шаг.
- **C:** Использовать XPF как основную стратегию. Лицензионные затраты + зависимость от вендора.

**Последствия выбора A:** Миграция займёт больше этапов, но каждый этап чистый и без vendor lock-in.
**Последствия выбора B:** Нужен spike-бюджет. Если XPF подойдёт — сэкономит Этапы 2-4.
**Последствия выбора C:** Лицензионные расходы, потенциальная зависимость.

**Рекомендуемый default:** A — чистая миграция.
**Что блокируется:** Ничего — но если выбор B, нужно начать spike до Этапа 2.
**Дедлайн решения:** До Этапа 1.

## D5 — Цель платформ Avalonia-приложения: только Windows или кросс-платформа?

**Статус:** Открытый

**Контекст:** Avalonia поддерживает Windows, macOS, Linux, WebAssembly. Текущее WPF-приложение — Windows-only. Решение влияет на:
- Использование Windows-only API (Registry, `AllocConsole`, `DllImport`)
- Упаковку и установщик
- Тестовую инфраструктуру
- Приоритет bug fixes

**Варианты:**
- **A (рекомендуемый):** Windows-only в первой итерации. Avalonia для возможности кросс-платформы в будущем. Минимальный scope.
- **B:** Кросс-платформа с первого Avalonia-релиза. Нужно устранить все Windows-only зависимости (P/Invoke, некоторые Redis/Kafka настройки).

**Что блокируется:** Windows-only API в `App.xaml.cs` (`AllocConsole`), installer. Если цель B — нужен spike по P/Invoke изоляции.
**Дедлайн решения:** До Этапа 4 (Avalonia bootstrap).

## D6 — `IDialogService`: синхронный или асинхронный?

**Статус:** Открытый

**Контекст:** Avalonia диалоги — асинхронные (`await dialog.ShowAsync()`). WPF MessageBox — синхронный. При создании общего `IDialogService` нужно выбрать интерфейс.

**Варианты:**
- **A (рекомендуемый):** Асинхронный интерфейс (`Task<bool> ConfirmAsync(...)`). WPF-реализация оборачивает в `Task.FromResult()`.
- **B:** Синхронный интерфейс. Avalonia-реализация блокирует thread (плохо).
- **C:** Оба перегруза. Усложняет интерфейс.

**Рекомендуемый default:** A — async-first. Соответствует Avalonia и modern .NET.
**Что блокируется:** PR 2.3 (замена MessageBox).
**Дедлайн решения:** До Этапа 2.

## D7 — SVG-иконки: переходить или остаться на PNG?

**Статус:** Открытый (Spike)

**Контекст:** Текущие иконки — PNG (60+ файлов, `pack://application:,,,/Icons/`). Avalonia поддерживает SVG через сторонние пакеты (`Avalonia.Svg`, `Svg.Skia`). PNG работает из коробки через `avares://`. SVG даёт масштабируемость и меньший размер.

**Варианты:**
- **A:** Остаться на PNG, перенести на `avares://`. Минимальный риск.
- **B:** Провести spike: оценить SVG-пакеты (Avalonia.Svg или Svg.Skia), преобразовать несколько иконок, проверить качество. Если подходит — перейти на SVG в Этапе 9.
- **C:** Смешанный подход: критичные иконки → SVG, остальные → PNG.

**Рекомендуемый default:** B (spike в рамках Этапа 9). Не блокирует ранние этапы.
**Что блокируется:** Окончательный дизайн Этапа 9.
**Дедлайн решения:** До Этапа 9.
