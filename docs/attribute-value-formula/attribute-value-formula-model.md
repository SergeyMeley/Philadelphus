# Значение атрибута и система формул (отдельная задача)

Самостоятельная задача, вынесенная из миграции на Avalonia в отдельную ветку. Цель — чистая
модель значения атрибута, где **формула — единственный источник истины**, а значение
вычисляется/кешируется. Здесь же собрано текущее состояние формульного UI (перенесено из
`docs/avalonia-migration/15-avalonia-tech-debt.md`, разделы B/C и формульные пункты E/I).

> Этот документ — единая точка по теме формул. В документацию миграции на Avalonia формульная
> тема больше не входит.

---

## 1. Целевая модель

- **`ValueFormula`** — единственное, что задаёт пользователь и что хранится канонически.
- **`Value`** — материализованный результат вычисления формулы. **Кеш**, нужен отчётам и
  экспорту. Пишет его ТОЛЬКО пересчёт; UI/таблицы `Value` напрямую не трогают никогда.
- **«Переопределено»** = локальная `ValueFormula` ≠ родительской. Один источник, конфликт
  `Value`/`ValueFormula` исчезает (с ним — и класс багов вокруг `_isValueOverridden`).
- Вычисление, граф зависимостей, материализация `Value`, режимы пересчёта — **бизнес-логика
  в Core** (доменный сервис). Presentation — только отображение и триггер правки.

## 2. Правила хранения формулы (что именно пишем)

Храним **ровно как задал пользователь**, без принудительной нормализации:

| Ввод | Хранимая `ValueFormula` |
|------|--------------------------|
| Выбор из выпадающего списка | `=[uuid]` |
| Введённая формула (`=[uuid]`, `=ЛИСТ("Мой лист")`, `="Мой лист"`, …) | дословно, как ввёл пользователь |
| Системный базовый тип, «голое» значение без `=` (`12000`, `abc`) | литерал-формула по типу: `=12000`, `="abc"`, дата/время — в формате типа |
| Пользовательский тип, «голое» значение без `=` | **отклоняем** (только пик или формула; ввод формул НЕ запрещаем) |

`Value` всегда = результат вычисления `ValueFormula`.

## 3. Отображение и редактирование

- **Просмотр ячейки (обе таблицы):** всегда вычисленное значение (или код ошибки формулы).
- **Строка формул:** дословная `ValueFormula`.
- **Редактирование ячейки значения:**
  - системный базовый тип — текстовый ввод, показываем **вычисленное** значение; «голый»
    ввод оборачиваем в литерал; ввод с `=` — формула дословно;
  - пользовательский тип — пик из списка (→ `=[uuid]`) или формула с `=`; «голый» текст как
    простое значение **не принимаем**.
- **Мгновенность:** правленый атрибут пересчитывается СРАЗУ и синхронно, независимо от режима
  пересчёта (Manual/Auto/AutoCurrentElement).

## 4. Архитектурный сдвиг

Уже в Core: `FormulaAstEvaluator`, `FormulaResult`, `FormulaExecutionContext`
(`Philadelphus.Core.Domain.FormulaEngine`).

Оркестрация (пересчёт с графом зависимостей и циклами, материализация `Value`,
форматирование ошибок) переезжает из presentation (`RepositoryFormulaBarVM`) в доменный сервис
проекта FormulaEngine. Presentation после переезда: строка формул и таблицы дёргают сервис и
показывают `Value`/код ошибки.

## 5. Шаги (с билдом между ними)

1. **Core-сервис пересчёта** — `IAttributeFormulaService` (+ реализация) в FormulaEngine: перенос
   оркестрации из `RepositoryFormulaBarVM` без смены поведения. **Сделано.**
2. **VM делегирует** в сервис (фабрика контекста + колбэк UI-оповещения), перенесённые методы
   из VM удалены. **Сделано.**
3. **Переопределение по формуле.** `_isValueOverridden`/`GetEffectiveValue` → сравнение
   `ValueFormula`. Удалить текущий workaround (пере-присвоение `Value`).
4. **Запись только формулой.** `AttributeValueText`/`EditText`-путь/WPF-сеттер перестают писать
   `Value`. Пик → `=[uuid]`, системный «голый» ввод → литерал, формула → дословно, «голый» текст
   для пользовательского типа → отказ. После записи — синхронный пересчёт атрибута.
5. **UX отображения/редактирования** по разделу 3 (обе таблицы + строка формул).
6. **Миграция данных.** Существующие строки (`Value` задан, формула пуста) → `=[uuid]`
   (пользовательский тип) / `=<литерал>` (системный). Делать раньше.
7. **Тесты** на каждый шаг.

## 6. Текущий прогресс

- **Шаги 1–2 сделаны:** оркестрация вынесена в `Philadelphus.Core.Domain.FormulaEngine/Services/`
  (`IAttributeFormulaService` + `AttributeFormulaService`); `RepositoryFormulaBarVM` делегирует
  (контекст — через `contextFactory`, UI-оповещение — через `onAttributeChanged`); перенесённые
  методы из VM удалены, `RecalculateFormulaAttribute` оставлен тонкой обёрткой.
- **Ранее сделано (UI значения атрибута):** общий `AttributeValueText`
  (`GetDisplayText`/`GetFormulaText`/`SetFormulaText` над `ElementAttributeModel`); каналы
  `DisplayTexts`/`EditText` в `ChildCollectionTableRow`; ячейка значения в таблице наследников
  идентична таблице атрибутов; фикс обнуления унаследованного значения при ручном вводе
  (пере-присвоение `Value` после очистки формулы — **временный workaround, снять на шаге 3**).
- **Дальше:** шаги 3–7.

Затрагиваемые файлы: `Philadelphus.Presentation/Services/Tables/AttributeValueText.cs`,
`ElementAttributeVM.cs`, `Philadelphus.Presentation/Models/Tables/ChildCollectionTableRow.cs`,
`ChildCollectionTableBuilder.cs`, `Philadelphus.Presentation.Avalonia/Behaviors/DataGridColumnsBehavior.cs`,
`RepositoryFormulaBarVM.cs`, `Philadelphus.Core.Domain.FormulaEngine/Services/*`, доменная модель
`ElementAttributeModel` (семантика переопределения), персистентность/маппинг значений.

## 7. Открытые вопросы / риски

- **Синтаксис литералов.** Обёртка системного «голого» ввода должна давать валидную формулу
  (строки — в кавычки с экранированием; даты/время — в ожидаемом формате). Уточнить токенайзер/парсер.
- **Производительность.** `Value` остаётся кешем (не вычисляем в геттере), отчёты читают кеш.
- **Загрузка.** `Value` персистится как кеш; на загрузке либо доверяем кешу, либо прогоняем пересчёт.

## 8. Состояние формульного UI в Avalonia (перенесено из тех-долга миграции)

### 8.1. Формульные взаимодействия с таблицами (бывший раздел B)

Строка формул (`FormulaBar`) и текст-бокс-behaviors (`FormulaBarTextBoxBehavior`,
`FormulaSuggestionTextBoxBehavior`) перенесены и работают; отложенная пересборка по уходу фокуса
(`DataGridLostKeyboardFocusCommandBehavior`) работает. НО три WPF-behavior'а, завязанных на ячейки
`DataGrid`, **не реализуемы публичным API Avalonia** (`DataGridCell.OwningColumn`/`OwningRow`/
`ColumnIndex` — internal; нет `CurrentCell`/`SelectedCells`/фокуса ячеек):

- `DataGridFormulaCellSelectionBehavior` — вставка ссылки на ячейку кликом (определение колонки/
  строки → `ChildFormulaCellSelection`).
- `DataGridFormulaReferenceHighlightBehavior` — подсветка выбранных ячеек-ссылок рамкой с фильтром
  «только атрибутные / колонка Значение».
- `DataGridCellFocusRequestBehavior` — возврат фокуса в конкретную ячейку колонки.

Варианты: (а) reflection к internal-членам (хрупко); (б) собственный контрол/тема ячейки с
публичными хуками; (в) альтернативный UX вставки ссылок (через окно редактора формул). Пока
формулы пишутся вручную в строке формул/редакторе.

### 8.2. Подсветка синтаксиса в строке формул (бывший раздел C)

**Сделано.** Подсветка наложена прямо в поле ввода: раскрашенный `TextBlock` (инлайны через
`FormulaHighlightOverlayBehavior.Segments`) лежит ПОД редактируемым `TextBox` с прозрачным текстом
при активной подсветке (`boolToBrushConverter` с `Transparent|` от `IsFormulaHighlightOpen`;
каретка — `CaretBrush`). Цвета тема-зависимы (`FormulaHighlightKindToBrushConverter`, светлая «VS» /
тёмная «VS Code Dark+», выбор по `Application.Current.ActualThemeVariant`). Совпадение переносов
оверлея и `TextBox` — одинаковые шрифт/размер/`Padding` и единый вес шрифта. Поле многострочное с
авто-переносом (`TextWrapping=Wrap`, `AcceptsReturn=False`). Парная скобка под курсором —
подчёркивание (фон инлайнового `Run` недоступен). Нюансы: фон `TextBox` переопределён на
`Transparent` во всех состояниях (иначе Fluent перекрывает оверлей при фокусе);
автодополнение — при префиксе ≥ 2 символов и подавляется внутри строкового литерала.

**Остаточный долг (тема):** держать подсветку синтаксиса тема-зависимой при «живой» смене темы
(сейчас перекрашивается при следующей пересборке сегментов).

### 8.3. Ячейка значения атрибута (бывшие пункты E/I)

- Редактируемый ввод — штатный editable `ComboBox` (Avalonia 11.3: `IsEditable`+`Text`), без
  `SelectedItem` (связка `SelectedItem`+`Text` сбрасывала значение в null при ручном вводе). Выбор
  из списка пишет «[uuid]» через `TextSearch.TextBinding`; ввод формул/значений — через `Text`.
- Общая логика — `AttributeValueText`; в таблице наследников каналы `DisplayTexts` (просмотр) и
  `EditText` (TwoWay-редактирование) в `ChildCollectionTableRow`; `DataGridColumnsBehavior`
  (Avalonia) в покое показывает `DisplayTexts[...]`, при редактировании — editable `ComboBox`,
  один-в-один с `Attributes.axaml`. Ячейка наследников идентична ячейке таблицы атрибутов.
- Известный временный workaround (снять на шаге 3): в `AttributeValueText.SetFormulaText` после
  `TrySetSystemBaseValueFromString` значение пере-присваивается последним действием, чтобы очистка
  `ValueFormula` не снимала признак переопределения у унаследованного атрибута.
